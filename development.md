# Directra 开发文档（产品思想版）

## 1. 文档目标

本文档用于定义 Directra 的产品思想、系统分层、模式设计与核心数据结构原则，作为后续开发、架构调整、交互设计和功能取舍的共同基线。

当前阶段，这不是技术实现文档，也不是任务排期文档，而是产品内核文档。

它要回答的核心问题是：

- Directra 到底是什么产品
- 它与普通 AI 视频生成工具的本质区别是什么
- 系统应该按什么层次来理解和构建
- 双模式交互为什么成立
- 哪些核心数据结构是平台的真正内核
- 为什么后续开发必须围绕项目状态与编辑工程展开

---

## 2. 产品定义

Directra 不是一个“AI 一键生成视频工具”，而是一个：

**双模式 Agent 视频制作平台**

= **自然语言入口 + 导演台调度 + Skill 驱动的领域工作流 + 长视频一致性系统 + Studio 编辑工程**

它的目标不是“生成一个视频”，而是：

**把一个模糊的视频想法，组织成一个可推进、可确认、可修改、可持续演进、可最终交付的视频项目。**

这意味着 Directra 的核心价值不在单次生成，而在：

- 项目理解
- 制作流程组织
- 领域化生产协议
- 长视频一致性维护
- 编辑器接管与局部重做
- 项目工程化保存与版本管理

---

## 3. 产品本质与边界

### 3.1 Directra 不是 Prompt 工具

用户进入 Directra，不是在写 prompt，而是在发起一个制作任务。

用户的一句话，表达的是项目意图，而不是一次模型调用。系统要先理解：

- 做什么类型的视频
- 目标受众是谁
- 主要目标是什么
- 预计时长是多少
- 希望的叙事方式和风格是什么
- 更适合走哪个领域 Skill
- 哪些节点需要用户确认
- 最终如何进入编辑与交付

所以 Directra 的输入不是 prompt，真正的输入是：

**项目意图。**

### 3.2 Directra 不是聊天机器人

Directra 里的 Agent 不是陪聊助手，而是导演台和调度系统。

它不是负责“陪用户对话”，而是负责：

- 理解项目
- 规划路径
- 选择 Skill
- 推进阶段
- 调用生成能力
- 停在需要确认的地方
- 把项目送入可编辑的工程状态

所以 Agent 的角色是：

**项目推进者，而不是对话本身。**

### 3.3 Directra 不是单次生成器

普通生成产品的目标是尽快产出一个结果。

Directra 的目标是：

- 先把需求变成项目
- 再把项目变成结构化内容
- 再把结构化内容变成可编辑工程
- 再让用户持续重构、精修、版本化

所以 Directra 是一个项目系统，而不是一次性结果系统。

---

## 4. 核心价值主张

用户真正需要的，不是单次的“能生成”，而是持续的“能掌控”。

Directra 要提供的价值包括：

### 4.1 降低视频制作的组织复杂度

用户不再需要自己协调从选题、脚本、分镜、素材、风格到剪辑的整条链路。

系统要承担“组织生产”的负担。

### 4.2 把模糊需求转成可执行项目

很多用户有想法，但没有制作结构。

Directra 要把模糊想法逐步转化为：

- brief
- 大纲
- 脚本
- 分镜
- 素材计划
- 时间轴工程

### 4.3 把自动化和可控性结合起来

系统不能只是自动生成，也不能把所有复杂度都丢给用户。

正确的模式应该是：

**Agent 主导流程，用户主导决策。**

### 4.4 支持局部修改，而不是强迫全部重来

用户必须能够：

- 只改某段脚本
- 只重做某个镜头
- 只替换某个片段
- 只改字幕和音乐
- 只对某一版做变体

这是平台从 demo 走向真实生产系统的关键。

## 5. 双模式设计

Directra 的交互核心不是单一页面，而是两个互补的工作模式：

- Agent 创作模式
- Studio 编辑模式

这两个模式不是两个独立产品，而是同一个项目生命周期中的两种主导方式。

### 5.1 模式 A：Agent 创作模式

适合从 0 到 1。

这是项目发起、方向确认、结构成型和初稿生成的主要模式。

它的特点包括：

- 聊天驱动
- 多轮确认
- 自动规划
- 自动出初版
- 以项目推进为核心，而不是即时输出为核心

它解决的是“不确定性”问题，也就是：

- 用户想法还不够清晰
- 制作路径还没确定
- 领域流程还没选择
- 项目结构还没形成
- 用户需要系统先帮忙组织

所以 Agent 创作模式的本质是：

**通过对话，把模糊想法组织成结构化项目。**

### 5.2 模式 B：Studio 编辑模式

适合精修和生产。

这是项目进入编辑工程后的主要模式。

它的特点包括：

- 时间轴
- 片段替换
- 字幕与音乐编辑
- 转场
- 局部重生成
- 工程保存
- 版本管理

它解决的是“确定性控制”问题，也就是：

- 初稿已经有了
- 用户想精修
- 用户只想改某一部分
- 用户要做多个版本
- 用户要管理素材、时间轴和导出

所以 Studio 编辑模式的本质是：

**把初稿转化为可精修、可版本化、可持续修改的视频工程。**

### 5.3 双模式的关系

这两个模式不是割裂关系，而是递进关系。

产品路径应该是：

1. 用户在 Agent 创作模式中发起项目
2. 系统通过导演台和 Skill 组织出初稿结构
3. 项目逐步形成脚本、分镜和时间轴草案
4. 用户进入 Studio 编辑模式进行工程化精修
5. 后续仍然可以在 Studio 中继续通过自然语言发起局部修改

所以两种模式共享同一个项目状态、同一套数据结构、同一条版本历史。

### 5.4 局部重生成的双轨原则

局部重生成必须是平台级能力，并且必须同时支持两种路径：

#### 路径 A：自然语言重构

用户直接说：

- 把第三个镜头变得更紧张一点
- 这一段口播更短更直接
- 这一段背景音乐换成更轻快的
- 这段转场更柔和一点

系统把自然语言解析为具体编辑操作或重生成任务。

#### 路径 B：人工重构

用户直接在 Studio 中手动操作：

- 替换素材
- 调整时长
- 拖拽片段
- 修改字幕
- 锁定片段
- 删除镜头
- 插入转场

所以“局部重生成”不应该只被理解为 AI 指令，而应该被理解为：

**系统允许用户通过自然语言或人工编辑两种方式，对项目的局部结构和局部结果进行重构。**

## 6. 产品工作流原则

Directra 的工作流不应该是“一轮提问，一轮输出”，而应该是一个分阶段推进的项目流程。

### 6.1 基本推进顺序

一个典型项目应当沿着这样的路径推进：

1. 项目意图理解
2. Skill 路由
3. 方向与 brief 确认
4. 大纲确认
5. 脚本生成与确认
6. 分镜生成与确认
7. 素材与片段生成
8. 一致性检查
9. 初稿工程组装
10. Studio 编辑接管
11. 局部修改与版本演进
12. 导出

### 6.2 确认节点原则

系统不能把所有内容一次性自动做完。

必须存在若干用户确认节点，例如：

- 方向是否正确
- 大纲是否成立
- 脚本是否可接受
- 分镜是否符合预期
- 风格是否统一
- 最终版本是否可以导出

这保证产品不会滑向“黑盒生成”。

### 6.3 先结构，后素材

在流程上，应优先生成高层结构，再生成底层素材。

原因是：

- 结构可确认
- 结构可修改
- 结构能驱动一致性
- 结构稳定后，生成层成本更低

因此建议的生产优先级是：

**brief -> outline -> script -> storyboard -> assets -> timeline**

## 7. 系统分层架构

Directra 可以按下面六层理解。

### 7.1 User Interface Layer

这一层不是简单的页面集合，而是用户对同一个视频项目的不同观察和控制界面。

应包括：

- Chat Panel
- Storyboard Panel
- Timeline Editor
- Assets
- Skill Config
- Project Explorer
- Preview
- Export

这一层的本质是：

**项目工作台。**

它既要支持 Agent 创作模式，也要支持 Studio 编辑模式。

### 7.2 Agent Orchestration Layer

这一层是导演台与任务调度中心。

可包含：

- Intent Agent
- Planner Agent
- Skill Router
- Review Agent
- Script Agent
- Storyboard Agent
- Asset Agent
- Edit Agent

这一层不是为了堆砌多个 Agent，而是为了把项目推进拆成明确职责。

它的本质是：

**把项目从模糊需求推进为结构化成果的调度层。**

### 7.3 Skill Runtime Layer

这一层负责承载领域工作流。

可包含：

- Built-in Skills
- User Skills
- Workflow DSL
- Constraints
- Templates
- Domain Rules
- Approval Gates
- Tool Binding

这一层是 Directra 平台战略价值的重要来源之一。

Skill 在这里不是 prompt，而是：

**一个领域化的生产协议。**

### 7.4 Long-video Consistency Engine

这一层负责维护长视频项目中的连续性。

可包含：

- Character Registry
- Story Bible
- Scene Memory
- Style Memory
- Audio Theme Memory
- Continuity Checker
- Shot Bridge Planner
- Transition Planner

这一层的本质不是增强生成，而是维护持续演化的项目世界状态。

### 7.5 Generation & Editing Layer

这一层负责执行具体生成和处理能力。

可包含：

- LLM
- Image Gen
- Video Gen
- TTS
- Music
- Subtitle
- Lip Sync
- Frame Interpolation
- FFmpeg
- Timeline Render

这一层是能力层，不是产品定义层。它应该服从上层结构，而不是决定产品形态。

### 7.6 Data & Infrastructure Layer

这一层是项目系统成立的基础。

可包含：

- PostgreSQL
- Redis
- Object Storage
- Vector DB
- Queue
- Analytics
- Auth
- Billing
- Project Versioning

它的意义不只是存储，而是确保项目具有：

- 持久化
- 可追踪
- 可回退
- 可版本化
- 可恢复

## 8. Skill 的产品意义

Skill 是 Directra 与一般 AI 视频工具拉开差距的关键机制。

### 8.1 Skill 不是模板

Skill 不应该只是“某个场景下的一段提示词”。

它应该定义：

- 这个领域的目标是什么
- 这个领域的视频结构如何组织
- 这个领域的脚本怎么写
- 这个领域的分镜怎么拆
- 哪些节点必须确认
- 哪些一致性最重要
- 默认优先调用哪些工具和资产

### 8.2 Skill 是领域工作流

Skill 的本质应当是：

**领域知识 + 工作流 + 约束 + 风格 + 资产绑定**

因此 Skill 的职责不是让内容“更像某种风格”，而是让整个制作过程“按这个领域的正确方式发生”。

### 8.3 平台与 Skill 的关系

平台负责通用能力：

- 项目管理
- Agent 调度
- 数据结构
- 一致性
- 编辑器
- 导出

Skill 负责领域化差异：

- 内容组织方法
- 脚本规则
- 分镜规则
- 用户确认点
- 风格偏好
- 工具优先级

这意味着：

**平台提供能力，Skill 定义做法。**

---

## 9. 长视频一致性思想

### 9.1 长视频的壁垒不是“更长的生成”

真正的难点不是输出更长时长，而是维护：

- 角色一致
- 场景一致
- 故事一致
- 风格一致
- 音乐一致
- 镜头过渡自然

### 9.2 长视频必须通过拆解组织

长视频不应被理解为一次性生成结果，而应拆解为：

**Act -> Scene -> Shot -> Clip**

然后通过：

- 独立 clip 生成
- 一致性上下文注入
- 桥接镜头
- 转场规划
- 编辑器组装

把它组织为完整作品。

### 9.3 一致性应独立成层

一致性不应依赖单个模型的偶然表现，而应被设计为独立系统。

这意味着平台要维护的不是“单轮输出”，而是一个持续演进的视频世界状态。

---

## 10. 编辑器的产品地位

编辑器不是附属功能，而是 Directra 的最终落点。

### 10.1 没有编辑器，平台就只是 demo

如果用户不能接管项目并持续修改，平台就只是在生成样例，而不是真正进入生产。

### 10.2 Studio 是项目控制中心

Studio 模式中，用户应能够：

- 调整时间轴
- 替换片段
- 增删镜头
- 重排结构
- 修改字幕
- 替换音乐
- 调整转场
- 锁定局部内容
- 发起局部重生成
- 管理多个版本

### 10.3 Chat 与 Timeline 必须互相映射

Studio 不是“只有拖拽”，也不是“只有聊天”。

正确方向是：

- 自然语言可以驱动编辑操作
- 手动编辑可以反映为结构化工程状态

所以 Chat 和 Timeline 应该是同一工程的两种控制界面。

## 11. 核心数据结构原则

Directra 的真正内核不是页面，也不是模型，而是数据结构。

后续开发应坚持：

**data model first**

也就是先定义项目对象、状态对象、内容对象和编辑对象，再围绕这些对象构建 Agent、Skill、UI 与基础设施。

### 11.1 为什么数据结构是内核

因为整个系统要靠这些对象来驱动：

- 项目推进
- 阶段确认
- 生成任务
- 一致性维护
- 时间轴组装
- 局部重做
- 版本回溯

如果这些对象定义不稳定，后续系统会变成页面逻辑、prompt 逻辑和模型返回之间的临时拼接。

---

## 12. 关键数据对象分层

以下对象不是最终数据库设计，而是产品内核应当具备的概念模型。

### 12.1 Project 系列

用于定义一个视频项目的总体状态。

应至少包含这些概念：

- `Project`
- `ProjectIntent`
- `ProjectGoal`
- `ProjectPhase`
- `ProjectState`
- `ProjectVersion`

这一组对象回答的是：

- 这个项目是什么
- 它想达成什么
- 当前进行到哪里
- 处于哪个模式
- 当前版本与历史版本是什么关系

### 12.2 Skill 系列

用于定义该项目应该按什么领域协议运行。

应至少包含这些概念：

- `Skill`
- `SkillProfile`
- `WorkflowDefinition`
- `ConstraintSet`
- `ApprovalGate`
- `ToolBinding`

这一组对象回答的是：

- 该项目走哪个领域
- 这个领域的流程是什么
- 哪些约束必须满足
- 哪些节点必须停下来确认
- 哪些工具优先使用

### 12.3 内容结构对象

用于定义作品本身的结构。

应至少包含这些概念：

- `Brief`
- `Outline`
- `Script`
- `Storyboard`
- `Act`
- `Scene`
- `Shot`
- `Clip`

这一组对象回答的是：

- 作品的逻辑结构是什么
- 每一段讲什么
- 每一镜怎么拍
- 每个片段如何组织

### 12.4 一致性对象

用于定义项目如何在长流程中保持稳定与连续。

应至少包含这些概念：

- `StoryBible`
- `CharacterRegistry`
- `CharacterState`
- `SceneMemory`
- `StyleMemory`
- `AudioThemeMemory`
- `ContinuityIssue`
- `TransitionPlan`

这一组对象回答的是：

- 这个作品的世界观和角色状态是什么
- 前后镜头是否冲突
- 风格是否漂移
- 音频主题是否延续
- 段落与段落如何自然衔接

### 12.5 编辑工程对象

用于定义 Studio 模式中的工程化状态。

应至少包含这些概念：

- `Timeline`
- `Track`
- `Segment`
- `AssetRef`
- `EditOperation`
- `RegenerationTask`
- `SubtitleTrack`
- `MusicTrack`
- `VersionSnapshot`

这一组对象回答的是：

- 初稿如何被组织成工程
- 素材如何挂接到时间轴
- 用户做了哪些编辑动作
- 哪些部分被局部重生成
- 当前工程与历史版本的关系是什么

---

## 13. 数据流与项目推进链

从产品角度，Directra 的一条主链可以理解为：

`Intent -> Project -> Skill -> Brief -> Outline -> Script -> Storyboard -> Timeline -> Edit -> Export`

与它并行存在的，是一条持续状态链：

`Story Bible / Character Registry / Scene Memory / Style Memory / Audio Memory / Version History`

第一条链负责内容推进。  
第二条链负责连续性和可控性。

两条链同时存在，才构成一个真正的视频项目系统。

---

## 14. 设计原则总结

后续开发调整时，必须反复回到以下原则：

### 14.1 做项目系统，不做 prompt 工具

用户发起的是视频任务，不是 prompt 请求。

### 14.2 做导演台，不做聊天助手

Agent 的本质是调度与推进，不是对话本身。

### 14.3 Skill 是领域工作流，不是风格模板

Skill 定义做法，而不是只定义语气。

### 14.4 长视频靠组织，不靠一次生成

通过结构拆解、一致性维护与编辑拼装来完成长视频。

### 14.5 编辑器是终点，不是补充

真正可用的产品必须让用户接管项目工程。

### 14.6 支持局部修改，不强迫全部重来

局部重构必须同时支持自然语言重构和人工重构。

### 14.7 数据结构是平台内核

后续架构和交互都应围绕核心对象展开。

## 16. 计费系统与多提供商架构

### 16.1 核心原则

Directra 必须做自己的积分/计费层，不能将 API key 暴露给用户，原因：

1. **成本控制**：视频生成成本极高，必须有用量上限与预扣机制
2. **多提供商统一**：用户不感知底层模型，平台统一调度
3. **可持续经营**：平台按消耗成本收费，保留合理利润空间

### 16.2 积分单位设计

**1 积分 = $0.001（千分之一美元）**

用户充值示例：$10 = 10,000 积分

各操作参考消耗：

| 操作 | 积分消耗 | 说明 |
|------|---------|------|
| 意图解析 / 多轮对话 | 1–2 | Claude 轻量调用 |
| 脚本生成 | 5–10 | Claude 中等长度输出 |
| 分镜文字生成 | 5 | Claude 结构化输出 |
| 图片生成（1张） | 20–40 | Flux / DALL-E |
| 视频生成（5秒） | 200–500 | Runway / Kling / Pika |
| TTS 语音合成 | 5–10 | ElevenLabs / OpenAI TTS |

### 16.3 计费流程

```
用户购买积分包（Stripe）
    ↓
每次操作前：检查积分余额，不足则拒绝
    ↓
操作执行时：预扣积分（防止并发超用）
    ↓
操作完成后：按实际 token/次 结算，退还多扣部分
    ↓
操作失败：全额返还预扣积分
```

**职责归属**：积分校验与扣减在 **Node BFF 层**完成，Python 执行层只负责执行任务，不处理计费。

### 16.4 生成器抽象层（多提供商）

Python 执行层需要一个统一的 `GeneratorService`，屏蔽底层差异：

```python
GeneratorService
  .generate_image(prompt, options)   # 自动路由：Flux / DALL-E / Midjourney
  .generate_video(frames, options)   # 自动路由：Runway / Kling / Pika
  .generate_speech(text, options)    # 自动路由：ElevenLabs / OpenAI TTS
```

设计目标：
- **成本路由**：优先调用单价低的提供商
- **降级容灾**：某服务不可用时自动切换
- **未来扩展**：支持高级用户绑定自己的 API key

### 16.5 开发阶段

计费系统列入 **Phase 9**，当前阶段不实现，但架构设计（数据库 schema、积分字段、操作日志表）应在 Phase 1（Prisma schema 设计）时一并规划占位，避免后期大改。
